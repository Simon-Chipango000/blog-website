Designing Resilient Systems: Circuit Breaker Pattern Implementation

In distributed systems, failures are inevitable. The Circuit Breaker pattern prevents a network or service failure from cascading to other services. Here's how to implement it effectively.

TypeScript Implementation 

class CircuitBreaker {
  private requestQueue: Array<() => Promise<any>> = [];
  private state: 'CLOSED' | 'OPEN' | 'HALF_OPEN' = 'CLOSED';
  private failureCount: number = 0;
  private successCount: number = 0;
  private nextAttempt: number = Date.now();
  
  constructor(
    private failureThreshold: number,
    private successThreshold: number,
    private timeout: number
  ) {}
  
  async call(service: () => Promise<any>): Promise<any> {
    if (this.state === 'OPEN') {
      if (this.nextAttempt <= Date.now()) {
        this.state = 'HALF_OPEN';
      } else {
        throw new Error('Service unavailable');
      }
    }
    
    try {
      const response = await service();
      return this.onSuccess(response);
    } catch (error) {
      return this.onFailure(error);
    }
  }
  
  private onSuccess(response: any): any {
    if (this.state === 'HALF_OPEN') {
      this.successCount++;
      
      if (this.successCount >= this.successThreshold) {
        this.close();
      }
    }
    
    this.failureCount = 0;
    return response;
  }
  
  private onFailure(error: any): void {
    this.failureCount++;
    
    if (
      this.failureCount >= this.failureThreshold ||
      this.state === 'HALF_OPEN'
    ) {
      this.open();
    }
    
    throw error;
  }
  
  private open(): void {
    this.state = 'OPEN';
    this.nextAttempt = Date.now() + this.timeout;
  }
  
  private close(): void {
    this.state = 'CLOSED';
    this.failureCount = 0;
    this.successCount = 0;
  }
}

This implementation provides a robust mechanism for handling service failures gracefully. The circuit breaker has three states: CLOSED (normal operation), OPEN (service unavailable), and HALF_OPEN (testing if service has recovered).

Database Sharding Strategies for High-Traffic Applications

As applications grow, database scalability becomes critical. Sharding distributes data across multiple databases to improve performance and enable horizontal scaling. Let's explore different sharding strategies and their trade-offs.

Key-based sharding uses a value from the data (like user ID) to determine the shard. Range-based sharding assigns ranges of values to different shards. Directory-based sharding uses a lookup table to locate data. Each approach has advantages depending on your access patterns and data characteristics.
